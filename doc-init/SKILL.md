# 文档体系初始化（doc-init）

执行分两个阶段：**先修复全局规范，再初始化项目文档**。

本文中的 `<DOC_INIT_DIR>` 指当前 `SKILL.md` 所在目录。执行脚本或读取 reference 时，先解析这个目录，不要写死某台机器的绝对路径。

---

## 内置脚本使用边界

优先用内置脚本完成机械动作，把模型上下文留给业务判断：

- `scripts/project_inventory.py`：扫描语言栈、构建文件、子模块、文档、配置、入口候选和多源证据候选；只产出候选事实，不决定业务域。
- `scripts/doc_coverage.py`：覆盖度闸门。把当前代码功能入口与已有领域地图的入口锚点做机械匹配，结合地图段的源码指纹基线戳，按退出码判定 `COMPLETE/STALE/NEEDS_INIT`，防止"旧地图存在就当已完成"的偷懒收工。Step 6.5 据其退出码分支。
- `scripts/upsert_agents_nav.py`：幂等新增或更新根 `AGENTS.md` 文档导航条目。
- `scripts/doc_nav_lint.py`：检查根 `AGENTS.md`、`CLAUDE.md`、`docs/` 导航一致性和常见生成偏差。
- `scripts/db_miner.py`：数据库 catalog 和领域级表/字段证据挖掘。
- `scripts/git_history_miner.py`：轻量挖掘 Git 历史弱信号，用于热点、历史叫法和 Q&A 线索。

脚本输出是证据和防呆，不替代模型对业务边界、主称谓、KB/Guide 粒度和落档内容的判断。

## 阶段一：校验并修复全局 AI 指令文件

### Step 1 — 定位全局所有 AI 指令文件的真身

探测以下文件，软链则跟到真身（`readlink -f`），对真身路径去重后得到待处理文件列表：

1. `~/.claude/CLAUDE.md`
2. `~/.codex/AGENTS.md`
3. `~/.codex/instructions.md`
4. `~/.config/opencode/AGENTS.md`

若以上均不存在，报告"未找到全局 AI 指令文件"并询问用户路径后继续。

### Step 2 — 脚本校验并自动插入/升级

对每个真身文件，直接运行脚本，脚本负责版本检测、旧章节替换和内容插入：

```bash
python3 <DOC_INIT_DIR>/scripts/insert_doc_governance.py "<真身路径>"
```

**根据脚本输出判断后续动作**：

| 输出前缀 | 含义 | 后续动作 |
|----------|------|----------|
| `[跳过]` | 已是最新版本 | 无需操作，直接进 Step 3 报告 |
| `[新增]` / `[完成]` | 首次插入成功 | 进 Step 3 报告 |
| `[升级]` / `[完成]` | 旧版已自动替换为新版 | 先做下方「散落旧规则清理」，再进 Step 3 报告 |

**仅在 `[升级]` 时需要模型额外清理**（脚本只替换章节本身，无法处理章节外的散落旧规则）：

扫描文件中是否还残留以下内容，有则删除：
- `## AGENTS.md 优先级` 整个章节
- `知识持久化` 章节下的 `### 检索在先、存储在后` 小节
- 其他以「文档放置」「文档索引」「AGENTS.md 导航」「docs/ 目录」为主题的散落段落

保留：`知识持久化` 章节的其余内容（禁用 memory 的各条规则）及所有与文档无关的章节。

### Step 3 — 输出阶段一报告

说明：
- 处理了哪些真身文件（去重后的路径列表）
- 每个文件：脚本输出是 `[跳过]` / `[新增]` / `[升级]`，以及清理了哪些散落旧内容（若有）

## 阶段二：初始化当前项目文档体系

Phase 2 的目标是按业务工作场景重建 AI Coding Agent 上手知识网络，而不是生成全局表/代码大索引。阶段二开始后先读取 `references/knowledge-network-design.md`，并用它控制文档粒度、命名、非业务项目处理和预算受限行为。

### Step 6 — 判断是否需要初始化 / 续接 / 复核

完成判定的**锚点**是根 `AGENTS.md` 的 `## 领域地图（doc-init）` 段，但「地图存在」只证明项目**曾经被初始化过**，不证明地图**仍覆盖当前代码**。老项目最常见的失败现场：地图是一两年前生成的，此后代码量翻倍、长出整块新领域，旧地图却被当成"已完成"直接退场——这是与"无地图却谎称已完成"对称的另一种偷懒。所以**凡检测到地图段，都必须先过 Step 6.5 覆盖度复核，复核通过才算完成**。

按以下优先顺序判断：

1. **根 `AGENTS.md` 已含 `## 领域地图（doc-init）` 段（无论有无 backlog 段）** → 先读地图段拿到已登记领域全集与各自状态，**进入 Step 6.5 覆盖度复核**，用当前代码校准地图后再决定收工还是续写。**禁止**因为"地图已存在 / 全是已生成 / 无 backlog"就直接退出。
2. **无 `## 领域地图（doc-init）` 段** → 不论 `docs/` 是否非空、根 `AGENTS.md` 是否已有「文档导航」章节或几条已登记的文档，一律判为**初始化未完成**，继续 Step 7/8 正常初始化；Step 8 建地图时要交叉比对已有文档并复用（见 Step 8），不是无视重写。

**反偷懒硬规则（两类对称偷懒都要堵）**：

- **谎称未完成式偷懒**：已有零散 KB/Guide、`docs/` 非空、根 `AGENTS.md` 已有「文档导航」——这些都**不是**初始化已完成的证据，完成判定的唯一锚点是 `## 领域地图（doc-init）` 段。
- **谎称已完成式偷懒（核心新增）**：地图段存在也**不是**收工的充分证据。地图只反映它生成那一刻的代码版图；必须经 Step 6.5 用当前 `project_inventory.py` 结果校准，确认地图仍覆盖当前代码的全部功能区且无明显漂移，才能收工。"地图里全是已生成、又没有 backlog"绝不等于"当前代码已被覆盖"——代码可能在地图生成后翻了一倍。
- `doc_nav_lint.py` 报出的 orphan-doc warning 数量再大，也**不能**被解释成"覆盖度合理的起点"用作收工或退避理由；它只是参考信号，不参与完成判定。

若根目录已有 `AGENTS.md` 或 `docs/`，可先运行一次导航检查辅助判断当前状态（仅供参考，不替代上面的判定树）：

```bash
python3 <DOC_INIT_DIR>/scripts/doc_nav_lint.py --root .
```

### Step 6.5 — 覆盖度复核（地图存在时强制，由脚本判闸，禁止凭自陈收工）

收工与否**不由模型自陈"看着覆盖得差不多"决定，而由 `doc_coverage.py` 的退出码判定**——这是本步存在的全部意义：把"旧地图是否仍覆盖当前代码"变成一个机械可算、模型无法绕过的硬闸门。

**第一步：重跑 inventory（代码可能已变，不复用旧 JSON），再跑覆盖度闸门**：

```bash
python3 <DOC_INIT_DIR>/scripts/project_inventory.py --root . --output .doc-init-project-inventory.json
python3 <DOC_INIT_DIR>/scripts/doc_coverage.py --root . --inventory .doc-init-project-inventory.json
```

`doc_coverage.py` 解析地图段的入口锚点和「覆盖度复核基线」戳，把当前代码的功能入口（`entry_candidates` + `submodules`）与锚点做前缀匹配，按下列任一条件判 `STALE`：入口覆盖率低于阈值、存在成片未登记功能区、代码量较基线增长超阈值、**或地图段根本没有基线戳（旧项目首次接入本机制时必然命中，强制做一次完整复核）**。退出码：`0=COMPLETE`、`2=STALE`、`3=NEEDS_INIT`。

**第二步：按退出码分支，禁止违背退出码收工**：

| 退出码 | 含义 | 动作 |
|--------|------|------|
| `3 NEEDS_INIT` | 实际没有地图段 | 回到 Step 6 判定 2，走完整初始化 |
| `0 COMPLETE` | 锚点覆盖当前代码、无明显增长 | 把脚本输出的 `建议写回地图段的基线戳` 写回地图段（刷新基线），告知用户「文档体系仍覆盖当前代码，整理压缩用 doc-compact、增量补充用 doc-update」后退出 |
| `2 STALE` | 覆盖不足 / 有未登记功能区 / 代码大涨 / 无基线戳 | **不算完成**，进入第三步 |

**第三步（仅 STALE）：把脚本给的机械事实落成业务判断与续写批次**。脚本只给"哪些功能区没被锚点覆盖"，不替你判定它是不是真实业务域：

1. **未覆盖功能区** → 逐个经产品北极星过滤（Step 7a 规则）：是真实功能则作为新领域行加入地图（标 `本次深写` 或 `待补充`）；是死代码/实现漂移则列待确认发现，不坐实成领域。
2. **代码量大涨 / 无基线戳触发的复核** → 对 `已生成（复用现有）` 领域做漂移点检（细则见 `references/scan-and-boundary-report.md`「覆盖度复核」一节）：抽查 KB 标注的代码/表/流程入口在当前代码里是否还落得到实处、锚点目录下是否冒出成片未覆盖新入口。明显漂移的领域从 `已生成（复用现有）` 降级为 `本次深写`（刷新），并在报告写明漂移点；轻微差异记录待 doc-update。
3. 然后回到 Step 7a → Step 8/9 深写本批；已确认仍准确的 `已生成` 领域不重扫、不重写。Step 9 收尾时把刷新后的基线戳写回地图段。

阈值默认（`--min-coverage 0.85`、`--max-uncovered-area-entries 3`、`--max-growth-pct 0.25`）适用于多数项目；纯库 / 脚手架等 `entry_candidates` 稀疏的项目可能因总入口数过低被判 STALE，此时读脚本的 `未覆盖功能区` 列表人工确认即可，不要靠调低阈值绕过闸门。

### Step 7 — 产品北极星先行 + 人机协同 Intake

读取 `references/human-intake.md`。

**Step 7a：先确立产品北极星，再做任何代码扫描**。doc-init 的常见失败模式是从代码反推产品形态：代码里的死代码、冗余功能、未接入导航的孤儿视图会被坐实成"业务领域"，建出与真实产品冲突的领域地图。代码只能定义实现拓扑（怎么实现的），不能定义产品形态（产品是不是真有这个功能）；产品形态以产品文档和用户确认为准。按顺序确立：

1. 读项目已有根 `AGENTS.md`，**跟随其产品指针**（哪怕指向跨仓的后端 PRD/路线图/北极星文档）。
2. 读用户提供或项目内的 PRD/北极星/路线图/设计稿。
3. 以上都没有 → 向用户征询一句话产品定义：做什么、给谁、核心循环。
4. 仍拿不到（用户已禁止提问）→ **硬卡**：停下，明确告知"缺产品权威来源，需要你一句话说清产品是什么才能继续"，不进入 Step 8 硬建领域地图。这是对下方"用户禁止提问不阻塞"原则的显式例外——产品真相是领域地图的地基，缺了它整张地图都不可信；区别于具体某个领域内的经验性 Q&A 空洞，那些仍按原则标低置信继续。

**确认真相后就地修正冲突文档（不止补缺）**：doc-init 不只创建缺失文档。任何阶段（产品北极星确认、Q&A、用户纠正）确立的真相，若与 `docs/` 已有文档（本次或更早 doc-init / 人工产物，不论是否已提交 git）相冲突，必须在当次会话内把冲突文档改对，不允许 `docs/` 同时留两份矛盾结论；按全局规范「§5 单一来源」执行。操作细则：

- **裁定（默认回权威源）**：默认回权威源（产品文档 / 后端 PRD / 代码）核实原文再定，不因旧文档"自称读过 PRD"就免检——这种自称本身是待验证断言。仅当用户看过源头后明确认为源头 / 代码本身已过时、坚持按最新认知来，经用户确认后以用户为准。
- **防拉锯（持久化裁定记录）**：用户推翻源头时，在对应 KB 写一条带日期和理由的【裁定】记录（格式见 `references/document-templates.md` §6），下次 doc-init / doc-update 读到它就不再翻案；同时提醒用户后端源文档也建议同步更新，但 doc-init 不擅自跨仓改。
- **传播（裁定后自动、不二次确认）**：真相一经裁定即自动改齐。改名（主称谓）在所有文档全量替换、保留实现别名（代码类名 / 表名照旧）；改逻辑——含删整篇 KB、抹领域地图行、改写交叉引用它的 KB——也自动执行。事后详列改了 / 删了什么；未提交文档被删后 git 无法恢复，删除清单必须完整。

**Step 7b：人机协同 Intake**。除非用户明确说「不要提问 / 直接生成 / 测试模式 / 干跑 / 只输出报告」，做轻量 Intake，收集资料入口、团队业务叫法、运行验证入口和老手经验。用户禁止提问时跳过本步，并在自评中标注「缺少用户经验输入」（注意：这条"不阻塞"豁免只适用于 7b 的经验性提问，不适用于 7a 的产品北极星硬卡）。

### Step 8 — 扫描项目并输出完整领域地图与知识边界报告

读取 `references/scan-and-boundary-report.md`。

**本步首要交付物顺序：产品北极星摘要 → 完整领域地图 → per-域详细报告**。产品北极星摘要（Step 7a 确立的产品定义来源、核心循环）先于地图给出；领域地图必须经产品北极星过滤——代码扫到的候选模块若在产品定义中找不到依据、且未接入导航/路由可达，标记为「候选死代码/实现漂移」列入待确认发现，不作为新领域写入地图。地图列出项目全部**真实**业务/功能领域，每个领域标注四种状态之一：「已生成（复用现有）」「本次深写」「待补充」「候选死代码/实现漂移」。

**建地图前先盘点已有文档，复用而不是无视重写**：交叉比对 `docs/` 现有 `*_KNOWLEDGE_BASE.md` / `*_GUIDE.md` 文件和根 `AGENTS.md` 已有「文档导航」条目。某个领域若已被现存文档覆盖（哪怕是旧版 doc-init 或人工写的），该领域状态直接标「已生成（复用现有）」，不重新生成、不重写；只有内容明显过期、与当前代码冲突，**或与本次产品北极星/用户确认的真相冲突**时才需要更新——后一种情况不是"可选优化"，必须修正，且要在报告中说明冲突点和修正前后的结论差异。

先运行项目结构 inventory，得到机械候选事实，减少重复读取和猜测：

```bash
python3 <DOC_INIT_DIR>/scripts/project_inventory.py --root . --output .doc-init-project-inventory.json
```

读取输出 JSON 后，再由 Agent 判断业务域、子模块边界、领域语言和隐藏机制；不要把脚本的 `entry_candidates` 当成最终文档结构。

读取 `references/multi-source-evidence.md`，结合 inventory 的 `evidence_sources` 做轻量多源证据发现：测试、接口契约、前端、配置、CI/CD、日志指标、迁移脚本、外部契约、权限字典、生成代码/元数据、运行时验证入口等只作为候选补强方向。不要在 Step 8 全量深挖所有证据源；具体深挖必须围绕候选业务域或公共机制进行。

若项目存在 Git 历史，读取 `references/git-history-mining.md`，再运行轻量弱信号扫描：

```bash
python3 <DOC_INIT_DIR>/scripts/git_history_miner.py --root . --output .doc-init-git-history.json
```

Git 历史只作为高噪声弱信号，用于发现热点风险、历史兼容线索、领域语言候选和 Q&A 追问；不得仅凭 commit message 生成当前业务结论。无 `.git`、浅克隆、空历史或命令失败时不阻塞，记录为「Git 弱信号不可用」。

若当前 Agent 环境提供 sub-agent / worker 能力，且用户没有限制预算、时间、文件数或禁止并行，可并行探索；否则由主 Agent 串行完成。使用 sub-agent 时必须给每个 worker 明确边界：扫描目标、最多读取文件数、必须输出的证据格式、不得写文件。最终由主 Agent 汇总、裁剪和写入文档。

识别语言栈后，按需读取语言专项隐藏语义扫描文档：
- Java/Kotlin：`references/hidden-semantics/java-kotlin.md`
- JavaScript/TypeScript：`references/hidden-semantics/javascript-typescript.md`
- Python：`references/hidden-semantics/python.md`
- Go：`references/hidden-semantics/go.md`
- C#/.NET：`references/hidden-semantics/csharp-dotnet.md`

扫描完成后输出知识边界报告，列出每个候选领域的业务范围、统一主称谓、代码入口、表入口、流程入口、用户/资料补充、隐藏机制、待 Q&A 空洞和公共 Guide 候选。

### Step 8.5 — 数据库证据挖掘（可选增强）

若项目存在数据库配置、用户提供只读连接，或候选业务域明显依赖数据库事实（状态、金额、余额、库存、租户、分表、流程、字典、配置等），读取 `references/database-mining/workflow.md`，使用 `scripts/db_miner.py` 作为 doc-init 内置数据库证据工具。

本阶段只做轻量 catalog：获取表清单、字段清单、主键/索引粗信息和注释，用于辅助业务域划分、表名前缀聚类、识别候选关键表和公共机制候选。不得在本阶段做全库 count、distinct、profile、sample-table 或 analyze-field。

完成 catalog / classify-catalog / plan-domain-scan 后，可调用 `scripts/db_miner.py summarize-catalog` 生成目录级摘要，用于知识边界报告；该命令只读取本地 JSON，不连接数据库、不读取行级数据。

数据库 catalog 只补强业务域知识网络，不生成全局数据库文档：
- 表和字段入口先作为候选写入知识边界报告。
- 跨领域共享的字典、配置、分表、流程元数据等机制进入公共 Guide 候选。
- 缺少连接、无法访问或用户禁止数据库挖掘时，在知识边界报告和自评中标注「缺少真实数据口径」。

默认按测试环境处理，数据库连接来源和值样本可以保留完整信息以便后续复用；只有用户明确要求脱敏或当前环境明确不是测试数据时，才给 `scripts/db_miner.py` 增加脱敏参数。

### Step 9 — 针对性 Q&A 与文档生成

读取 `references/human-intake.md` 中的精准 Q&A 规则，只问代码看不到但会影响 AI 改代码成败的问题。本步若 Q&A 或用户纠正确立了与已有文档冲突的逻辑或名词，按 Step 7a「确认真相后就地修正冲突文档」处理，不要把矛盾留到下次。

生成具体领域 KB 前，按当前领域从多源证据候选中选择性深挖相关资料：测试断言、接口契约、前端菜单/表单、配置开关、迁移/初始化脚本、日志/指标、外部契约、权限字典、生成代码或运行时证据。深挖结果只写入当前领域 KB 或公共 Guide；不要生成全局 `TEST_INDEX.md`、`API_INDEX.md`、`DDL_INDEX.md` 等资源索引。

生成具体领域 KB 前，若已有数据库 catalog，可针对当前业务域调用 `scripts/db_miner.py plan-domain-scan`；只有对当前领域的主表、明细表、状态表、配置表或代码复杂 SQL 直接读写表，才进一步调用 `sample-table` / `analyze-field`。关键表逐字段发现写入 KB 的「表与字段入口」「隐性约束」「AI 易错点」。

生成具体领域 KB 前，先统一该领域的业务语言：为每个核心概念选一个主称谓，正文稳定使用主称谓；代码类名、枚举名、表名、接口字段和数据库注释作为实现别名保留在入口索引或首次出现的括号中。默认不要生成独立术语表；只有同名异义、异名同义或参数易混会导致 AI 写错代码时，才在「核心业务规则与隐性约束」中写短小消歧。无法判断主称谓或冲突会影响需求沟通时，必须向用户确认；用户禁止提问时标低置信度。

读取 `references/document-templates.md`，**按本次深写主批**逐域生成：
- `docs/<DOMAIN>_KNOWLEDGE_BASE.md`
- `docs/<TOPIC>_GUIDE.md`
- 项目根 `AGENTS.md`
- 项目根 `CLAUDE.md`

每生成一份长期文档后立即回写根 `AGENTS.md` 导航，避免留下孤儿文档。根 `AGENTS.md` 不得引用用户级或全局指令文件，例如不要写 `@~/.claude/CLAUDE.md`。项目根 `CLAUDE.md` 只能规范化为单行 `@AGENTS.md`。

回写根 `AGENTS.md` 导航时，优先使用脚本保证幂等：

```bash
python3 <DOC_INIT_DIR>/scripts/upsert_agents_nav.py --root . --path docs/<DOMAIN>_KNOWLEDGE_BASE.md --when-to-read "<任务触发句>"
```

`--when-to-read` 触发句要覆盖该领域全部任务类型（修改/新建/评审/分析/排查/优化），不要只写「改 X 前」，评审/分析类任务否则不触发。范式见 `document-templates.md`。

**主批写完后，把所有待补充领域全部登记进 backlog 段**（不得静默丢弃）：

```bash
python3 <DOC_INIT_DIR>/scripts/upsert_agents_nav.py \
  --root . --backlog \
  --name "<领域名> KB" \
  --anchor "<入口目录>" \
  --when-to-read "<触发场景>"
```

每个待补充领域调用一次，幂等、可重复执行。

**最后，把完整领域地图持久化进根 `AGENTS.md` 的 `## 领域地图（doc-init）` 段**——这是 Step 6 完成判定的锚点，**即使本次没有 backlog（已全部覆盖）也必须写这一段**，否则下次运行 doc-init 会因为找不到地图段而重新误判成"未完成"。

写法：模型直接整段重写该 markdown 段（覆盖式幂等，不依赖脚本）。地图段开头必须带一行**覆盖度复核基线戳**，记录本次校准时的源码指纹，供下次 Step 6.5 做代码量基线对比；指纹值取自本次 inventory（`scan.scanned_files`、各 `languages[].file_count`、`submodules` 数量）和 `git rev-parse --short HEAD`（无 git 则省略提交）。格式：

```markdown
## 领域地图（doc-init）

<!-- 覆盖度复核基线：2026-06-21 · 源码指纹 扫描 1573 文件 / Go 412 · TS 88 / 11 子模块 · 基线提交 a1b2c3d -->

| 领域 | 入口锚点 | 状态 |
|------|---------|------|
| 渠道体系 | src/channels/ | 已生成（复用现有） |
| Agent 执行循环 | src/agents/ | 本次深写 |
| LLM Provider 体系 | packages/llm-runtime/ | 已生成（复用现有） |
| 插件体系 | src/plugins/ | 待补充 |
| 存储/状态 | src/storage/ | 待补充 |
```

只跑了 Step 6.5 覆盖度复核、判定 `G + R == 0` 直接收工时，仍要把这一行基线戳按当前指纹刷新写回（地图领域行不变），让"上次复核到哪个代码版本"有据可查。

地图段、文档导航段、backlog 段三者必须一致：「已生成」类领域应同时出现在文档导航；「待补充」类领域应同时出现在 backlog；地图段是这两者的权威全集索引。

### Step 10 — 运行验证与 Operations 条件生成

只有项目存在本地运行、启动、验证或排障价值时，读取 `references/operations-validation.md`。

`OPERATIONS_GUIDE.md` 优先来自实践验证，而不是静态代码推断。用户允许执行时，尽量完成最小闭环：命令发现、构建验证、启动验证、存活验证、最小无副作用请求、日志排查、阻碍闭环、停止清理。用户禁止执行或环境不足时，只输出运行假设和待验证项。

运行验证产生的证据必须按语义分流：启动命令、探活、配置生效、日志路径和环境阻碍进入 `OPERATIONS_GUIDE.md`；业务接口真实行为、真实 SQL、状态变化、权限、缓存、MQ、错误码和链路副作用进入对应领域 KB；跨领域共享机制进入 `*_GUIDE.md`。不要把运行时看到的所有内容都塞进 Operations。

生成条件：
- 有至少一条可复用运行经验、验证命令、启动阻碍或环境差异 → 生成或更新 `docs/OPERATIONS_GUIDE.md`。
- 未执行运行验证但项目存在明显运行面 → 只生成薄版「运行假设与待验证清单」，所有静态推断标低置信度。
- 项目没有本地运行面且无通用验证价值 → 不生成 `OPERATIONS_GUIDE.md`，把验证方式写进根 `AGENTS.md` 或对应 Guide。

### Step 11 — 自我评估报告

自评前先运行文档导航一致性检查：

```bash
python3 <DOC_INIT_DIR>/scripts/doc_nav_lint.py --root .
```

将 lint 的 error / warning 纳入自评。可接受的 warning 必须说明原因；error 应先修复再报告完成。

**覆盖度账目（必须明确给出数字）**：

- 领域地图总数：N 个
- 已生成（复用现有）：A 篇（列出文件名）
- 本次深写：M 篇（列出文件名）
- 候选死代码/实现漂移：D 个（列出领域名 + 待确认原因）
- backlog 登记：N-A-M-D 个（列出领域名）
- 断言：A + M + D + backlog 数 = 地图总数 N ✓（若等式不成立，说明存在静默丢域，必须补录 backlog）
- 断言：根 `AGENTS.md` 的 `## 领域地图（doc-init）` 段已写入且与上述账目一致 ✓（缺失则必须补写，否则下次 Step 6 会误判成"未初始化"）
- 断言：地图段已带「覆盖度复核基线」戳，指纹为本次 inventory 当前值 ✓

**覆盖度复核账目（凡走过 Step 6.5 必须给出）**：

- 复核触发原因：地图段已存在（续接 / 疑似完成）
- 源码指纹：上次基线 → 当前（扫描文件数 / 子模块数 / 基线提交的变化）；无旧基线戳则注明"按可能严重过期做了完整复核"
- 当前功能区 C 个；地图已登记 M 个；已覆盖匹配 X 个
- 覆盖缺口 G 个（列名 + 锚点 + 已转 本次深写/待补充/产品北极星待确认）
- 疑似过期需刷新 R 个（列名 + 漂移点 + 已刷新/已转 doc-update）
- 断言：G + R == 0 才允许判定"真正完成"；否则本次必有对应深写或 backlog 登记 ✓

输出文档覆盖度自评：
- 每份领域 KB 是否覆盖业务范围、代码入口、表入口、流程入口、深层机制、隐性约束、验证路径、关联文档。
- 领域语言覆盖：每个核心概念是否已统一主称谓；是否保留必要实现别名；是否仍存在同名异义、异名同义或高风险参数混用。
- Git 弱信号覆盖：是否存在可用 Git 历史；是否只做轻量扫描；热点路径、历史叫法和 fix/revert/兼容线索是否只进入待确认或 Q&A；是否避免把 commit message 写成权威业务规则。
- 数据库证据覆盖：是否连接成功；是否只做 catalog；哪些业务域有数据库 catalog 证据；哪些业务域进一步做了 sample-table/analyze-field；哪些关键表仍未逐字段分析；哪些字段是低置信推断；缺失数据库证据的原因。
- 多源证据覆盖：哪些证据源可用；哪些只做了轻量发现；哪些领域已深挖测试/接口契约/前端/配置/迁移/日志/外部契约/权限字典/生成元数据/运行时证据；哪些高价值证据缺失或被判定噪音。
- 高风险未覆盖项：概念名相似但未消歧、状态流转原因不明、机制只看到调用但不知道生效条件、验证路径缺失等。
- Operations 验证覆盖：已执行哪些构建/启动/探活/最小请求/日志检查；哪些仍是低置信假设；不生成 Operations 的原因。
- 后续沉淀建议：哪些内容应由 `doc-update` 补入 KB/Guide，哪些文档适合后续由 `doc-compact` 整理压缩。
