# 文档整理与压缩（doc-compact）

执行时**先探测当前环境实际生效的文件，不要硬编码路径**——本文出现的 `~/.claude/CLAUDE.md`、`~/.config/agentsync/AGENTS.md` 等都是示例。

`<DOC_INIT_DIR>` 指 `doc-init` skill 所在目录，默认与本 skill 同级：`<本 SKILL.md 所在目录>/../doc-init`。doc-compact 与 doc-init 共同维护同一套 AGENTS.md/CLAUDE.md/docs 结构，凡是 doc-init 已有权威实现的检查或安装动作（全局规范安装、反向全局引用检查、docs 自我导航检查、领域地图/backlog 状态），doc-compact 一律调用 `<DOC_INIT_DIR>/scripts/` 下的脚本，不重写一份相近但可能跑偏的逻辑。**`<DOC_INIT_DIR>` 不存在时**（doc-compact 被单独分发，没有附带 doc-init）：跳过相应调用，改为人工核对，并在报告里明确标注"未找到 doc-init，本项未自动校验"——不要因为找不到就直接跳过不提。

## §0 标准：这套文档管理模式（判定基准）

doc-compact 自身审计/压缩时使用的判定基准。下面是精简 checklist；**每条的完整说明、心智模型和易变事实细则见 [`references/standard.md`](references/standard.md)**，拿不准时按需读取。

**重要**：§0 是 doc-compact 自己的检查清单，**不是"待插入全局文件的内容"**。全局指令文件里真正应安装/比对的权威文本，唯一来源是 doc-init 的 `<DOC_INIT_DIR>/scripts/insert_doc_governance.py`；§0 若多出本该全局生效的规则，提给 doc-init 维护者合并进 STANDARD 升版本号，不要在这里自行写入全局文件。

**心智模型**（详见 reference）：纵向三层（规则层 `AGENTS.md` → 索引层 `<DOMAIN>_INDEX.md` → 内容层）+ 横向类型（开放集合，常见五类作默认，可新增并登记）。

1. **唯一一级入口**：导航集中在根 `AGENTS.md`「文档导航」；禁止裸 `OVERVIEW.md`/`INDEX.md` 散索引；合法二级索引具名 `<DOMAIN>_INDEX.md`。
2. **最多两跳直达**：核心一跳、次要经 `<DOMAIN>_INDEX.md` 两跳；禁三级以上嵌套（特例见 §3）。
3. **只列真实文档**：一行一个带一句话用途；不写空占位、历史/元注释。
4. **描述写成「何时该读」**：回答「带着什么问题 / 改哪块代码时该读它」。
5. **inline 指针为主、底部表兜底**：指针就近贴在它支撑的规则旁。
6. **CLAUDE.md 只能一行 `@*.md`**：不承载内容；无独立约定的模块不单设 AGENTS.md/CLAUDE.md。
7. **文档放置与命名**：各类型有固定目录；`.md` 不放源码目录（`AGENTS.md` 例外但需构建排除）。完整放置表以全局指令文件为准。
8. **增删改文档 → 同步索引**：同步根 `AGENTS.md` 导航与全仓相对链接。
9. **两级索引两条独立触发**：① 规模驱动；② 类型驱动（排查/Review 台账 ≥3 篇折叠）。详见 §3。
10. **规范且精简**（见 §4 / §5）：命名/链接/语言合规、导航按领域聚类；删不改变行为的冗余，留改变动作/判断的内容。
11. **易变事实不跨文档复述（单一来源）**：易变数字只在权威源维护；其他文档要么链接、要么只写不变的结论句。**反模式**：把所有含该数字的文档都改成链接（制造过度耦合）。细则见 reference。

目标环境若已用别的措辞表达了等价规则，视为合规，不必改写。

## 流程

Step 1 校验全局规范 → Step 2 只读审计 → Step 3 判定拆二级索引 → Step 4 修复结构 → **Step 5 整理压缩（核心交付，不可跳过）** → Step 6 验证。

## Step 1 — 校验全局指令文件

1. 定位实际生效的全局文件：Claude Code 的 `~/.claude/CLAUDE.md`、Codex 的 `~/.codex/AGENTS.md` 等；若是 `@import` 或软链，**跟到真身再改，别改软链壳**。
2. **`<DOC_INIT_DIR>` 存在时**：对每个真身文件直接运行 `python3 <DOC_INIT_DIR>/scripts/insert_doc_governance.py <真身路径>`——这与 doc-init 自己 Step 2 用的是同一个脚本，doc-compact 不维护独立的安装文本副本。根据输出前缀判断后续动作：`[跳过]` 已最新；`[新增]`/`[完成]` 首次插入成功；`[升级]`/`[完成]` 旧版已自动替换（脚本只替换它认识的 `## 项目文档管理` 整章，不动其它内容）。脚本写入完成后，人工扫一遍全局文件其余章节，找是否还散落着早于该脚本存在、且与刚写入内容主题重复的文档放置/索引/导航类段落（常见于脚本接入前手写的旧约定）；若有，按其规则是否已被新写入的章节覆盖逐条判断，覆盖了就删，未覆盖的规则先确认是否该提给 doc-init 维护者合并进 STANDARD。
3. **`<DOC_INIT_DIR>` 不存在时**（doc-compact 单独分发）：退回手工模式——逐条对照 §0，标出缺失或过时项（尤其残留的 `OVERVIEW.md`/`INDEX.md` 旧索引模式）；缺失则把 §0 对应内容补进全局文件，过时则就地更正；并在报告中注明"未找到 doc-init，全局规范安装文本为手工比对结果，建议后续安装 doc-init 以获得版本化管理"。

## Step 2 — 只读审计（先扫描，不改）

运行 `scripts/audit.sh [项目根]`（默认当前目录；只读，自动排除第三方 / 构建产物 / 备份 / `.git` / `.claude` worktree），一次跑完机器可判的检查：

- **A** CLAUDE.md 是否都只有一行 `@*.md`（任意单行 `@` 引用，如 `@AGENTS.md`、`@~/.claude/CLAUDE.md`）
- **B** 悬空 `@AGENTS.md`（引入但同级无 AGENTS.md）
- **C** 裸 `OVERVIEW.md`/`INDEX.md`（散索引文件或引用）与工具注入块残留；具名 `<DOMAIN>_INDEX.md` 合法，不报
- **D** 各 AGENTS.md 行数（> 500 行 → Step 3）
- **E** 孤儿文档：docs/ 与 specs/ 下未被「根 AGENTS.md ∪ 任意 README.md ∪ 任意 `*_INDEX.md`」引用的 .md。**E 必须 union 所有二级索引文件**，否则做了两级索引的项目会把全部二级文档误报成孤儿——脚本已处理，`*_INDEX.md` 自身也跳过检查。
- **F** 文件命名合规（排查记录 `YYYY-MM-DD-*.md`、review 台账 `*-review.md`、文件名不含空格）。
- **G** 预置折叠建议（故障排查 / Review 台账 ≥3 篇但未折叠），建议性，不计入小结成败。
- **H** doc-init 联动检查（`<DOC_INIT_DIR>` 存在时调用其 `doc_nav_lint.py`）：反向全局引用、docs 内自我导航残留、各项目根 `## 领域地图（doc-init）`/`## 待补充知识库（doc-init backlog）` 是否存在——**结果直接影响 Step 4/5 的保护边界，见下方「受保护的 doc-init 记账段」**。`<DOC_INIT_DIR>` 不存在时该节跳过并提示三项需人工核对。

脚本判不了的靠人读：文档放置是否错位（散落源码目录 / 项目根）、依赖文档的规则旁是否缺 inline 指针、是否冗余膨胀（→ Step 5）、**是否存在易变事实跨文档复述**（§0 条 11）——快速扫法：`grep -rn "具体数字 / 版本号 / 档位标识" docs/ specs/`，命中超过 2 处即疑似复述，逐一判断是权威源还是该删的副本。

### 受保护的 doc-init 记账段

若检查 H 显示某项目 `domain_map_present=True` 和/或 `backlog_present=True`，该项目根 `AGENTS.md` 里的 `## 领域地图（doc-init）` 与 `## 待补充知识库（doc-init backlog）` 两段是 doc-init 自己续接初始化进度的唯一依据（删除会让 doc-init 误判该项目从未做过领域覆盖，重新触发全量初始化）。Step 4 重建索引结构、Step 5 压缩内容时，**这两段原样保留，不删除、不折叠进二级索引、不当作"导航重复"精简掉**——可以挪动位置（如挪到文件更靠后处），但段标题与表格内容不改写。

若某项目根 `AGENTS.md` **没有**「## 领域地图（doc-init）」段（`domain_map_present=False`）：doc-compact 不负责补这段（那是 doc-init 的职责范围），只在 Step 6 报告里向用户提一句"该项目可能未完成 doc-init 领域覆盖初始化，本次审计只处理结构卫生，不代表领域文档已完整"。

## Step 3 — 判定是否拆二级索引（仅大项目）

**默认单层平铺**（根 `AGENTS.md` 一张表直达全部）。**能不做就不做**——每多一跳，模型多一次「要不要钻进去」的判断，漏读概率累加，所以小项目拆了更差。**这条保守只约束「是否拆二级索引」这一项结构决策，不约束 Step 5 的压缩**——压缩默认要做，不得借「能不做就不做」跳过整理压缩。

二级索引有两条独立触发逻辑，满足其一即应折叠：

**① 规模驱动**
- 导航篇幅 ≳ `AGENTS.md` 的 1/2，或规则被导航挤到文件后半（主判据）；
- 兜底硬阈值：`AGENTS.md` > 500 行且导航占相当篇幅（对应检查 D）。
- 拆法：根只保留「任务域索引入口」（每域一行，描述写「做 X 类任务时进来找」）；明细下沉到各域具名 `<DOMAIN>_INDEX.md`（**不沉到 README**）；二级聚类按「会带着什么问题来找」划分，跨类文档在相关聚类都可达。

**② 类型驱动（预置折叠，与项目大小解耦）**
- 故障排查记录（`docs/troubleshooting/YYYY-MM-DD-*.md`）**≥ 3 篇** → 应折叠到 `docs/troubleshooting/TROUBLESHOOTING_INDEX.md`，根留一条强路由。
- Review 台账（`docs/reviews/**/*-review.md`）**≥ 3 篇** → 应折叠到 `docs/reviews/REVIEW_INDEX.md`，根留一条强路由。
- 强路由必须包含「何时跳过 / 是否权威来源」，不能只是干瘪文件名；范例见全局规范「两级索引」节。
- 对应 Step 2 检查 G（建议性）：检测是否达折叠门槛但未折叠，以及折叠后根条目是否为强路由格式。

## Step 4 — 修复结构

- **CLAUDE.md**：非单行的，正文已在同级 AGENTS.md 有备份 → 改回单行 `@AGENTS.md`；只剩注入块、无自有内容 → 连同悬空 CLAUDE.md 一并删。
- **文档放置与命名**：错位 `.md` 移到该去的目录；文件名按所在类型对齐规范（知识库 / 指南 `SCREAMING_SNAKE_CASE`、设计 / review `kebab-case`、排查 `YYYY-MM-DD-*`）。移位 / 改名都要搜全仓引用同步更新——属破坏性，先列清单确认。
- **规范化**（机械、近确定，可直接做）：跨文档引用统一改相对路径；非 README 文档混入英文正文纠正为中文；标题层级修正（单 H1、不跳级）；导航每条描述对齐「何时该读」句式。
- **导航描述逐条审查（重点，不能跳过）**：逐条读根 `AGENTS.md` 文档导航里的每条描述，判断它是「何时该读」句式（回答「带着什么任务/问题来时读它」）还是「它讲了什么」句式（罗列文档内容）。后者路由效果差，一律改写为前者。改写标准：触发动作要覆盖全部任务类型（修改/新建/评审/分析/排查/优化），不罗列文档结构。全部条目审完后再做结构性操作。**反模式——任务覆盖不全（最易漏）**：描述只有修改动词（「改 X 前」），缺评审/分析/排查/优化类触发词；对天然会有评审/分析任务的领域（UI、设计、业务规则、性能等）一律补全。
- **索引结构重建**：只列真实文档、一行一句「何时该读」；删死链 / 空占位，补漏收文档；**条目按领域聚类、高频在前排序**，不按追加顺序堆。预置折叠类型（故障排查 / Review 台账）若已达 3 篇门槛，将平铺条目替换为对应强路由条目；强路由须含「何时跳过 / 是否权威源」。`## 领域地图（doc-init）`/`## 待补充知识库（doc-init backlog）` 两段不算导航条目，重建索引时跳过，保持原样（见 Step 2「受保护的 doc-init 记账段」）。
- **inline 指针**：给依赖文档的规则就近补指针，底部表保留兜底。
- **注入块 / 旧索引**：删裸 `OVERVIEW.md`/`INDEX.md`（这类与根竞争的散索引）与工具注入块。删注入块的操作步骤：① 读块内容，识别其中有哪些规则/约定（如 Always Do、Never Do、Resources 表、CLI 映射表）；② 逐条对照根 `AGENTS.md`，若某条已在 AGENTS.md 对应章节覆盖 → 直接跳过；③ 若有未覆盖的规则 → 提炼为原生中文规则语言，并入 AGENTS.md 对应章节（如"代码导航"、"核心约定"）；④ 全部内容处理完毕后，**删除整个注入块（含 `<!-- .*:start -->` 到 `<!-- .*:end -->` 之间的所有内容及标签本身）**；⑤ 不要在 AGENTS.md 里加"副作用说明"或"注意事项"作为替代——注入块要么内容已覆盖（直接删），要么内容提炼进正文后删。具名 `<DOMAIN>_INDEX.md` 是合法二级索引，保留不动。
- 过大 → 按 Step 3 拆。

## Step 5 — 整理压缩：本 skill 的核心交付（结构修好后做）

**反偷懒硬规则（不可静默跳过）**：

- 整理压缩是 doc-compact 存在的理由，不是结构修复之后的可选尾步。范围内每一篇 `docs/`/`specs/` 文档都必须被**逐篇**过一遍压缩判据，不允许「整体扫一眼觉得没问题」就跳过整篇文档。
- 收工汇报必须给出**逐篇账目**：每篇文档「压缩前 N 行 → 压缩后 M 行」，或明确写「已审，无可压，原因：xxx」。一篇文档都没改动、且没有逐篇「无可压」说明 = 判定为本次漏做压缩，不算完成。
- 判不准某句话该不该删时，先按下面的判据走一遍，仍判不准才进入「高风险待确认」清单——不要因为犹豫就整篇跳过不审。

### 压缩判据

一句话判据：**这句话会改变读者的动作或判断吗？不会就删。** 正确但不驱动行为的话是负资产，照删。

**删这些**（都不改变行为）：

- 交叉引用 / 「另见 X」之类不影响当前执行的链接。
- 复述：重复 frontmatter / 上文 / 标题已说过的内容。
- 重复提醒：同一条纪律在多处反复出现 → 收敛到一处。
- 死动机：不影响任何判断的解释从句、客套、「众所周知」「需要注意的是」。
- 历史沿革：「以前叫 X」「原本是…已废弃」「不再单设…」→ 属 git log。
- 空占位：「暂无」、空 TODO。

**留这些**（改变动作 / 判断，删了就失真）：

- 数字 / 阈值、标识符（类名 / 集合 / 字段 / 接口 / 枚举 / 配置 key）。
- 边界条件与例外、不变量、安全与数据约束、外部契约。
- **能翻转某个决策的那一句动机**（如「能不做就不做」配一句为什么，让模型倾向不拆）。只留这种会改判断的理由，纯背景理由照删。

**去重**：同一事实多处 → 留信息最全的一处（跨文件的通用规则只在全局留一份），其余删或改为一句指向；确认是同一事实（相同数字 / 边界 / 例外）才算重复。`## 领域地图（doc-init）`/`## 待补充知识库（doc-init backlog）` 两段不参与压缩判据，原样保留（见 Step 2「受保护的 doc-init 记账段」）。

**易变事实专项处置**（§0 条 11）：发现某具体数字散落多处时，先判断每处文档的角色：
- 权威源文档 → 保留，确认是唯一维护该数字的地方。
- 研究 / 评估 / 规格文档 → 不要改成链接，而是**提炼出不随该数字变化的结论句后删掉数字**（例：`2000+` 档不够 → 删数字，改为「需更高积分档，接入前核验」）。
- 如果发现自己在把多个文档都改成「→ 链接到权威源」，立即停下来——这是在制造耦合，不是在治理文档；重新判断这些文档本来需不需要知道这个数字。

**简化**：长句拆短、并列要点改表 / 列表、把绕的写直。

### 执行分级（直接做 vs 先确认）

- **低风险，直接删并在汇报里列出，无需逐条确认**：上面「删这些」列出的全部六类（交叉引用 / 复述 / 重复提醒 / 死动机 / 历史沿革 / 空占位）。这些都是 git 可回滚、明显不改变行为的内容，等用户逐条点头只会拖慢真正该做的事。
- **高风险，仍先列「拟删 / 拟改清单」给用户确认再动**：删整篇文档、成段重写、合并 / 拆分索引结构、删改含数字 / 阈值 / 标识符 / 边界条件的正文、易变事实的跨文档处置改动、文档内重排。
- 拿不准属于低风险还是高风险 → 归高风险，先问。

### 压缩后标识（必须，作为「本轮已审」回执）

范围内**每篇**文档处理完后（含判定为「无可压」的），在文件末尾追加或更新一行：
`<!-- 该文档整理/压缩于 YYYY-MM-DD -->`

- 若文件末尾已有同类标识 → 仅更新日期。
- 若末尾无标识 → 追加一行，与正文间留一空行。
- 标识行本身不参与压缩判据（不因它"不改变行为"而删除）。
- 这行标识同时是 Step 6 核对「有没有漏审」的凭据：范围内文档若没有本轮日期的标识，视为漏审——Step 6 的 `audit.sh --compact-date <今日>` 会逐篇机械核对该标识，缺失即报错。

### 文档内重排

按这份文档是什么类型、读者怎么用它，自行判断顺序该不该调。多数文档原结构已合理，默认不动；只有结构明显妨碍取用时才重排——属高风险动作，动正文前先列清单确认。

## Step 6 — 验证

重跑 `scripts/audit.sh <项目根> --compact-date <今日 YYYY-MM-DD>`（收尾必须带 `--compact-date`，启用检查 I 压缩标识硬闸门），确认：CLAUDE.md 全单行、无悬空 `@AGENTS.md`；无旧索引 / 注入块残留（第三方除外）；根 `AGENTS.md` 导航零死链零孤儿、docs/ 顶层散文件全收录；文件命名零违规；拆分后根 `AGENTS.md` 明显瘦身且二级索引完整；检查 H 的 `domain_map_present`/`backlog_present` 与改动前一致（不应因本次审计由 True 变 False）。

**压缩交付核对（核心，不可省略）**：
- **检查 I 必须 `压缩缺标识=0`**：脚本会逐篇列出 docs/specs 范围内缺本轮日期标识的文档。任何被列出的文档 = 本轮漏审，必须补审（压缩或确认无可压并补标识）后重跑，不得带着 `压缩缺标识>0` 收尾。这是逐篇账目从「模型自陈都审过了」升级成的机器闸门。
- 压缩 / 重排后逐项核对「会改行为的内容」仍在，未被误删。
- 汇报中给出 Step 5 的逐篇压缩账目（每篇前/后行数，或「已审无可压」说明）。

最后向用户报告改了什么、在哪、剩余风险与需手动操作的步骤；若项目缺「## 领域地图（doc-init）」段，额外提示该项目可能未完成 doc-init 领域覆盖初始化。

## 安全边界

- **分级处理**（与 Step 5「执行分级」一致）：复述 / 死链 / 历史沿革 / 空占位 / 重复提醒等低风险压缩可直接执行，在汇报里列出即可，不必逐条确认；删整篇文档、成段重写、拆 / 并索引结构、删改含数字 / 边界条件的正文，仍先列清单求用户确认。
- **不裁定内容真相**：doc-compact 只管结构卫生与压缩，无真相来源。发现两份文档对同一件事结论矛盾（内容冲突，非索引 / 放置 / 命名问题）时，**不自行判定谁对、不删任一方**，在报告里记一条并转 `$doc-init`（初始化 / 续接时）或 `$doc-update`（收工复盘时）按全局规范「§5 单一来源」裁定修正。
- **不碰**：第三方 / vendored 项目、构建产物、备份目录、git worktree。
- **工具注入会回来（需周期性复跑）**：代码情报类工具（如 gitnexus）在重新分析时可能把注入块写回来；这是 **"注入块当前仍须删除"** 的理由——注入回来了就再删，不是"注入块可以留着"的豁免。治本要在该工具侧关闭注入选项。
